# Spec 0001: MVP — Family Financial Manager

## Status

Draft — 2026-05-01

## 1. Purpose & Scope

A two-user web application for tracking household finances between Riza and Shima. Server-rendered, low-cost to operate, append-only ledger, multi-currency-aware via envelopes ("Pos"). Designed to be operated long-term with infrequent maintenance.

In scope:

- Two seeded users with shared visibility of all data.
- Telegram OTP login via an external AI-assistant API.
- Bank-account-only money tracking (other holdings represented as Pos).
- Envelope budgeting via "Pos," supporting reallocations, internal lending with debt tracking, and per-Pos custom currencies.
- Manual transaction entry (web UI) and programmatic entry (LLM-callable JSON API).
- A small set of views: filtered transaction list, current balances, Pos breakdown, spending-by-Pos over time.
- In-app notifications when the *other* user creates a transaction (per-user feed, persisted, with a list view and unread count).

Out of scope (deferred):

- Cash, e-wallets, credit cards, investment accounts as first-class Account types.
- Cross-currency exchange transactions (e.g., selling gold for IDR in one entry).
- Automatic FX-aware net-worth reports.
- Recurring/scheduled transactions.
- Attachment uploads (receipts, statements).
- Manual adjustment / write-off transaction type for explicit FX-loss tracking.
- Push notifications via Telegram beyond OTP (in-app persistent notifications exist; see §4.5, §6.5).
- Notification types beyond collaboration pings on transaction creation (target reached, Pos went negative, security alerts, etc.).
- Counterparty-total reports, monthly cash flow, net worth snapshots — see §6.

## 3. Users & Authentication

### 3.1 Users

Two users — Riza and Shima — are pre-seeded into the database. There is no self-service registration. Adding or modifying users requires editing the seed file and redeploying (acceptable since user changes are essentially never).

Each user record carries:

- `display_name` — `"Riza"` or `"Shima"`. Passed verbatim to the OTP assistant API.
- `telegram_identifier` — Telegram numeric ID **or** `@username`. Used as the login lookup key.

Both users see all data. No per-user privacy or access controls.

### 3.2 Login flow

1. User opens `/login`. Form has one input: Telegram ID or `@username`.
2. App looks up the matching user. If none, render `"User not found."` Do not disclose which identifier matched or didn't, beyond the lookup result.
3. App generates a 6-digit numeric OTP, stores it server-side keyed to the user with a 5-minute expiry, and increments a per-user OTP counter for rate limiting.
4. App calls the OTP assistant (§7.3): `POST /api/webhook/claude` with `{"message": "send OTP <code> to <display_name>"}`.
5. App renders the OTP-entry form. User receives the OTP via Telegram (assistant handles delivery) and enters it.
6. App verifies the OTP. On success, issues a session cookie valid for 7 days; clears the stored OTP. On failure, increment a per-OTP attempt counter; lock the OTP after 3 failed attempts (user must request a new OTP).

### 3.3 OTP defaults

| Parameter | Value |
|---|---|
| Length | 6 digits |
| Expiry | 5 minutes |
| Attempts before lockout | 3 |
| Resend cooldown | 60 seconds |

Flagged for review: tighten or relax independently of MVP design.

### 3.4 Sessions

- Cookie-based, 7-day rolling expiry on each authenticated request.
- `Secure`, `HttpOnly`, `SameSite=Lax`.
- Session token is a randomly generated opaque string stored in DB; logout revokes the row.

## 4. Data Model

### 4.1 Account

Represents a real bank account holding IDR. All accounts are IDR-denominated for MVP.

Fields:

| Field | Type | Notes |
|---|---|---|
| `id` | uuid | |
| `name` | text | e.g., `"BCA Shima"`. Free text, non-empty, displayed verbatim. |
| `archived` | boolean | Soft-delete; archived accounts hidden from default views, retained in history. |
| `created_at` | timestamptz | |

No `currency` field — IDR is implicit. No `opening_balance` field — opening balance is a transaction (see §10).

### 4.2 Pos

Represents an envelope / allocation bucket. Holds a balance in a single currency (default IDR; optionally any user-defined unit such as `gold-g`, `USD`).

Fields:

| Field | Type | Notes |
|---|---|---|
| `id` | uuid | |
| `name` | text | e.g., `"Tabungan Mobil"`. Free text, non-empty, unique per currency. |
| `currency` | text | Default `IDR`. Lowercase alphanumeric + dash; validated `^[a-z0-9\-]+$`. |
| `target` | numeric | Optional. Target balance for savings goals. May be `null`. |
| `archived` | boolean | Soft-delete. |
| `created_at` | timestamptz | |

Pos has three derived balances per currency, all computed from transactions (no stored balance):

- `cash_balance` — money currently allocated to this Pos.
- `receivables` — total amount this Pos is owed by other Pos (from inter-Pos `borrow` transfers, in destination currency terms — see §4.3).
- `payables` — total amount this Pos owes other Pos.

Negative `cash_balance` is permitted and first-class; it represents a Pos in deficit, resolved later via reallocation or future income (§5.4).

### 4.3 Transaction

The unit of all balance change. Append-only at the storage level (§5.2). Three types:

#### Type A: `money_in`

Money enters an Account from the outside world; allocated to one Pos.

| Field | Type | Notes |
|---|---|---|
| `id` | uuid | |
| `type` | enum | `money_in` |
| `effective_date` | date | Default today. Past dates allowed. Future dates rejected. |
| `account_id` | fk | Receiving account |
| `account_amount` | numeric | IDR amount entering the account. Always IDR. |
| `pos_id` | fk | Receiving Pos |
| `pos_amount` | numeric | Amount entering the Pos, in the Pos's currency. |
| `counterparty_id` | fk | Required. Source of funds (e.g., `"Salary"`). |
| `note` | text | Free text, optional. |
| `source` | enum | `web` \| `api` \| `seed`. Origin of the entry. |
| `created_by` | fk users | The authenticated user who created it (or `null` for `seed`/`api`). |
| `idempotency_key` | text | Unique. Used to dedupe API submissions. |
| `created_at` | timestamptz | Insert time. Distinct from `effective_date`. |
| `reverses_id` | fk transactions | Optional. Set when this is a reversal entry. |

For IDR Pos: `account_amount == pos_amount` (UI auto-fills; user enters once). For non-IDR Pos: user enters both — `account_amount` is the IDR exchanged, `pos_amount` is the unit tracked.

#### Type B: `money_out`

Money leaves an Account to the outside world; debited from one Pos. Same fields as `money_in` with sign reversed semantically.

#### Type C: `inter_pos`

Money moves between Pos. No Account involvement. Many-to-many: multiple source Pos, multiple destination Pos. Two modes via the `mode` field.

| Field | Type | Notes |
|---|---|---|
| `id` | uuid | |
| `type` | enum | `inter_pos` |
| `mode` | enum | `reallocation` \| `borrow` |
| `effective_date` | date | Same rules as A/B. |
| `note` | text | |
| `source` | enum | |
| `created_by` | fk users | |
| `idempotency_key` | text | |
| `created_at` | timestamptz | |
| `reverses_id` | fk transactions | |
| (line items, separate table) | | See below |

Line items live in `inter_pos_lines`:

| Field | Type | Notes |
|---|---|---|
| `transaction_id` | fk | |
| `pos_id` | fk | |
| `direction` | enum | `out` (source) \| `in` (destination) |
| `amount` | numeric | In this Pos's currency. |

**Cross-currency handling:** lines on opposite directions may carry different currencies. Each line is denominated in the line's Pos's own currency. The user enters all line amounts; the system does not convert.

**Borrow-mode debt tracking (Pattern P from interview):** when `mode = borrow`, the obligation is in the **destination's** currency. For each `out` line and each `in` line, a `pos_obligation` row is generated:

| Field | Type | Notes |
|---|---|---|
| `transaction_id` | fk | The borrow transaction that created this obligation. |
| `creditor_pos_id` | fk | The `out`-direction Pos (lent funds). |
| `debtor_pos_id` | fk | The `in`-direction Pos (received funds). |
| `currency` | text | Always the debtor's currency. |
| `amount_owed` | numeric | Computed: debtor's `in` amount, prorated across creditors when many. |
| `amount_repaid` | numeric | Accumulator. |
| `cleared_at` | timestamptz | Set when `amount_repaid >= amount_owed`. |

Repayment is another `inter_pos borrow` transaction in the reverse direction; the system matches against open obligations FIFO by `creditor_pos_id` / `debtor_pos_id` pair. If a repayment has no matching open obligation, it succeeds and creates a new obligation in the opposite direction (the creditor now owes the debtor). This handles the "kid's school cash short after gold price drop" case naturally — kids school is left at negative cash; resolution via §5.4.

**Reallocation mode** does not generate `pos_obligation` rows. The transfer is permanent.

### 4.4 Counterparty

A named external party for `money_in` / `money_out` entries.

| Field | Type | Notes |
|---|---|---|
| `id` | uuid | |
| `name` | text | Validated `^[a-zA-Z0-9_\- ]+$`, trimmed, no tabs/newlines. |
| `name_lower` | text | Lowercase index for case-insensitive match (autocomplete + dedup). |
| `created_at` | timestamptz | |

Autocomplete in the UI suggests existing rows; typing a new name creates a row inline. First-typed casing is preserved for display; later typing of the same lower-cased name resolves to the existing row.

### 4.5 Notification

Persistent in-app notification record. Per-user feed (each user has their own unread/read state). For MVP only one type exists (`transaction_created`); the schema is shaped to admit more types later.

| Field | Type | Notes |
|---|---|---|
| `id` | uuid | |
| `user_id` | fk users | Recipient. |
| `type` | enum | `transaction_created`. Other types deferred. |
| `title` | text | Short headline, e.g., `"Riza logged a money_out"`. |
| `body` | text | Optional detail (e.g., the amount, Pos, counterparty). |
| `related_transaction_id` | fk transactions | Nullable. When set, the UI links to `/transactions/:id`. |
| `read_at` | timestamptz | Null = unread. Set when the user marks it read. |
| `created_at` | timestamptz | Insert time. |

**Recipient rule** (single source of truth; lives in the Logic layer as `notification.RecipientsFor(tx, allUsers)`):

- For `source = seed`: no notifications. Initial setup never pings.
- For `source = web`: notify all users where `user_id != tx.created_by`.
- For `source = api`: `created_by` is null, so notify all users.

The function is pure and unit-tested; both web and API handlers depend on it.

## 5. Operations

### 5.1 Create

All three transaction types are creatable via:

- Web UI (HTMX form posts).
- LLM-callable API (§7.2).

Both paths share validation logic in the Logic layer (§8). Validation rules:

- `effective_date` ≤ today.
- `account` (where applicable) is non-archived.
- `pos` (where applicable) is non-archived.
- `pos.currency == line.amount.currency`.
- `counterparty` matches regex, trimmed.
- `account_amount` and `pos_amount` are positive numbers (sign comes from transaction type).
- For `inter_pos`: at least one `out` line and one `in` line; `Σ(out)` and `Σ(in)` are tracked per-currency separately (no cross-currency totals required).
- For `money_in` / `money_out` with IDR Pos: `account_amount == pos_amount`.
- `idempotency_key` is unique (silent success on duplicate).

### 5.2 Edit

Fields fall into two edit modes:

| Field | Mode |
|---|---|
| `amount` (account/pos/line) | reverse-and-re-enter |
| `effective_date` | reverse-and-re-enter |
| `account_id` | reverse-and-re-enter |
| `pos_id` | reverse-and-re-enter |
| `counterparty_id` | hard-edit |
| `note` | hard-edit |
| `type` / `mode` | non-editable (delete + recreate) |

**Reverse-and-re-enter** = the system inserts a `reverses_id`-pointing reversal transaction with sign-flipped amounts, then inserts a fresh corrected transaction. The original row is retained. Three rows total.

**Hard edit** = `UPDATE` the row in place. No reversal entry.

### 5.3 Delete

Logical delete only. The system inserts a reversal transaction. The original row is retained. No `DELETE` SQL on the `transactions` table is ever issued by application code.

### 5.4 Notification side-effects

Every transaction insert (web or API) writes notification rows in the **same database transaction** as the underlying ledger entry. If the notification write fails, the transaction insert rolls back — there is never a state where a transaction exists without its notifications, or vice versa.

Edits trigger notifications on the *modification* (the reversal entry, the re-entry, or the hard-edit), not on the original record. Hard edits to `note` and `counterparty` are subtle changes; they still produce a notification so the other user knows something changed.

### 5.5 Negative balances

A Pos with `cash_balance < 0` is a normal state. The UI surfaces it (red badge, list filter), but the system imposes no automatic remediation. The user resolves by:

- Inter-Pos reallocation from a positive Pos.
- Allocating future income (`money_in` to that Pos).
- Acknowledging the deficit (do nothing).

No "manual adjustment" or "write-off" transaction type exists in MVP.

## 6. Views

Four views. All authenticated; both users see the same data.

### 6.1 Transaction list (`/transactions`)

Filterable list of all transactions, newest first. Filters:

- Date range (default: last 30 days).
- Account (multi-select).
- Pos (multi-select).
- Counterparty (multi-select).
- Type (`money_in` / `money_out` / `inter_pos`).

Each row shows: date, type, amount (per-currency), account, Pos(es), counterparty, note. Reversal entries display with a strikethrough or "reversed" badge linking to the row they reverse.

### 6.2 Current balances (`/`)

Home view. Two sections:

- **Accounts** — table of non-archived accounts with current IDR balance, sorted by name.
- **Pos** — table of non-archived Pos grouped by currency. For each: name, cash balance, receivables, payables, target (if set), progress-toward-target percentage. Negative-cash Pos surfaced with a marker.

### 6.3 Pos breakdown (`/pos/:id`)

Single-Pos detail view: name, currency, target, current cash / receivables / payables, list of open obligations (creditor or debtor), and a chronological transaction list scoped to this Pos.

### 6.4 Spending by Pos over time (`/spending`)

Bar or table view: rows are months, columns are top-N Pos by money_out volume. Date range selectable, default last 6 months.

### 6.5 Notifications (`/notifications`)

Per-user feed. List view, newest first, with unread rows visually distinguished from read.

Each row shows: title, body, relative timestamp (`"2 minutes ago"`), and links to `/transactions/:related_transaction_id` when set. Two actions: click a row to mark it read; "Mark all read" button at top.

Unread count surfaces in the global header as a badge on a bell icon, refreshed via HTMX poll on a 30-second interval (cheap, no WebSocket needed for two users). Clicking the bell opens `/notifications`.

Out of MVP: cash flow per month, counterparty totals, multi-currency net-worth.

## 7. APIs

### 7.1 Web routes (HTMX)

Server-rendered HTML, partials returned for HTMX swaps. Echo group `/` (no prefix). Authenticated via session cookie; unauthenticated requests redirect to `/login`.

### 7.2 LLM API (JSON)

Echo group `/api/v1`. Authenticated via header `x-api-key: ${LLM_API_KEY}`; rejected with `401` on missing or invalid key.

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/accounts` | List accounts. |
| `POST` | `/accounts` | Create account. |
| `GET` | `/pos` | List Pos. |
| `POST` | `/pos` | Create Pos. |
| `GET` | `/counterparties` | List counterparties (for autocomplete-equivalent on the LLM side). |
| `POST` | `/transactions` | Create a transaction (any type). |
| `GET` | `/transactions` | List with filters mirroring §6.1. |
| `POST` | `/transactions/:id/reverse` | Insert a reversal entry (used by edit/delete flows). |

All requests and responses are JSON. Errors return `{"error": "code", "message": "..."}` with appropriate HTTP status. All POST endpoints accept an `idempotency_key` field; duplicate keys return the original record with `200 OK`.

Validation: identical to the web UI. The Logic layer is the single source of validation truth (§8); both API and web handlers call the same functions.

Every transaction created via this API is stamped `source = api`.

### 7.3 OTP delivery (outbound)

The app calls a single external endpoint to deliver login OTPs.

- **URL**: `${OTP_ASSISTANT_URL}/api/webhook/claude` (env-configured; default to the operator's instance).
- **Headers**: `x-api-key: ${OTP_ASSISTANT_API_KEY}`, `Content-Type: application/json`.
- **Body**: `{"message": "send OTP <code> to <display_name>"}`.
- **Timeout**: 5 seconds.
- **Retries**: none. On error, log and surface `"Failed to send OTP. Try again."` to the user.

Both API keys (`LLM_API_KEY`, `OTP_ASSISTANT_API_KEY`) live in environment variables and are never committed to source control.

## 8. Code Structure

Per CLAUDE.md §5, the source tree splits into three layer-folders. Cycles between Logic folders are resolved with a shared `common/` Logic folder.

### Logic — pure, deterministic

- **No external dependencies.** No `database/sql`, no Echo, no `time.Now()`, no `rand`, no globals.
- **Receives time and randomness as parameters** (`Clock`, `IDGen` interfaces).
- **Encodes all domain invariants and validation rules** (§5.1, §10).
- **Unit-tested with property-based tests for every named invariant** (§10).
- **Theoretically extractable** to a separate Go module without touching the rest of the app.

Concrete contents (proposed):

- `logic/money` — IDR-cents type, currency strings, arithmetic.
- `logic/pos` — Pos rules, balance computation from transactions.
- `logic/transaction` — transaction types, validation, reversal generation.
- `logic/obligation` — borrow-mode debt tracking, repayment matching.

### Dependencies — generated and external

- **`db/`** — sqlc-generated Go code. Migrations live in `db/migrations/`, queries in `db/queries/`, sqlc config at the repo root.
- Anything else that wraps an external system (the OTP assistant client) lives here.

### I/O — glue

- **`web/`** — Echo handlers, `html/template` setup, HTMX partials. Both human web routes and the LLM JSON API live here, since both are I/O.
- **`web/handler/`** — handler functions per route group.
- **`web/template/`** — `*.gotmpl` templates.
- **`web/middleware/`** — auth, logging, recovery.

Top-level: `cmd/server/main.go` wires everything together; `config.go` reads env vars.

## 9. Initial Setup

Day-zero balances enter the system as actual transactions (Pattern III from interview) — there are no "opening balance" fields on Account or Pos.

Workflow (operator-side):

1. Operator points an LLM at the existing Google Sheet of historical balances.
2. LLM calls `POST /api/v1/accounts` for each account in the sheet.
3. LLM calls `POST /api/v1/pos` for each Pos (with currency, optional target).
4. LLM calls `POST /api/v1/transactions` once per opening-balance entry, type `money_in`, counterparty `"Opening Balance"`, with `effective_date` set to whatever date the sheet records as the opening date. `source = api` automatically.

Per-currency invariant `Σ(Pos.cash) = Σ(Account)` is verifiable after seeding by reading current balances; no app-side enforcement at insert time beyond §5.1.

## 10. Correctness Invariants

Reinforced from CLAUDE.md §7. Each is encoded in the Logic layer and tested with a property-based test.

1. **Money is integer cents.** All `numeric` columns store smallest unit (rupiah for IDR; smallest gram subdivision the operator chooses for `gold-g`). Never `float64` in Go.
2. **Logic layer is deterministic.** No `time.Now()`, no `rand`, no globals. `Clock` and `IDGen` interfaces are injected.
3. **Append-only ledger.** No application-issued `DELETE` or balance-mutating `UPDATE` on the `transactions` table. Corrections are reversal + re-entry. Hard-edits (note, counterparty) are scoped to non-balance-affecting columns and verified by a test that asserts no balance-affecting column is `UPDATE`d.
4. **Idempotency on every state-changing op.** Every `POST` (web and API) carries an `idempotency_key`; the unique constraint on `transactions.idempotency_key` enforces it.
5. **Per-currency reconciliation.** For every currency `c`: `Σ(Pos.cash where currency=c) = Σ(Account where currency=c)` after every transaction. (In practice IDR-only on the Account side; non-IDR currencies live entirely in Pos and reconcile to themselves via inter-Pos balance.) Tested as a property-based assertion over generated transaction streams.
6. **Inter-Pos line totals.** For each `inter_pos` transaction and each currency `c` represented in its lines: `Σ(out lines where currency=c) = Σ(in lines where currency=c)`. (Cross-currency lines are not summed; each currency reconciles independently.)
7. **Borrow obligations close cleanly.** For every `pos_obligation` row, `cleared_at` is set if and only if `amount_repaid >= amount_owed`. No partial limbo state.
8. **Notifications and transactions are atomic.** A transaction row exists if and only if all of its notification rows exist. Enforced by the DB transaction wrapping both writes (§5.4) and tested with a fault-injection test that fails the notification write.
